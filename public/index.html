Cache-Control headers en servidor
Indica al navegador cuánto tiempo guardar cada recurso localmente
Alto impacto

Sin Cache-Control, el navegador descarga todos los recursos en cada visita. Con estos headers, imágenes, CSS y JS se guardan localmente por 1 año — la segunda visita de un usuario carga hasta 80% más rápido.

nginx.conf Copiar 
# Agregar dentro del bloque server{} en tu nginx.conf

location ~* \.(jpg|jpeg|png|gif|webp|svg|ico|css|js|woff2|woff|ttf)$ {
    expires           1y;
    add_header        Cache-Control "public, immutable";
    add_header        Vary "Accept-Encoding";
    access_log        off;
}

# HTML: siempre revalidar (para que los cambios apliquen)
location ~* \.html$ {
    add_header        Cache-Control "no-cache, must-revalidate";
}
.htaccess Copiar 
# Pegar en tu archivo .htaccess (raíz del sitio)

<IfModule mod_expires.c>
  ExpiresActive On
  ExpiresByType image/webp              "access plus 1 year"
  ExpiresByType image/jpeg              "access plus 1 year"
  ExpiresByType image/png               "access plus 1 year"
  ExpiresByType text/css                "access plus 1 year"
  ExpiresByType application/javascript  "access plus 1 year"
  ExpiresByType font/woff2              "access plus 1 year"
  ExpiresByType text/html               "access plus 0 seconds"
</IfModule>

<IfModule mod_headers.c>
  <FilesMatch "\.(jpg|jpeg|png|webp|css|js|woff2)$">
    Header set Cache-Control "public, immutable"
  </FilesMatch>
</IfModule>

Preload del elemento LCP
Indica al navegador que cargue primero el elemento más importante de la página
Alto impacto

El LCP (Largest Contentful Paint) es la imagen o bloque de texto más grande visible al cargar. Si no le dices al navegador que lo priorice, lo descubrirá "tarde" — después de procesar todo el HTML. Con rel="preload" le damos una ventaja de 0.5 a 1.5 segundos.

HTML <head>Copiar
<!-- Pegar ANTES de cualquier otro link/script en el <head> -->

<!-- Si el LCP es una imagen: -->
<link
  rel="preload"
  as="image"
  href="/img/hero-banner.webp"
  fetchpriority="high"
>

<!-- Si el LCP es una fuente: -->
<link
  rel="preload"
  as="font"
  href="/fonts/brand-bold.woff2"
  type="font/woff2"
  crossorigin
>

<!-- Para Shopify, en layout/theme.liquid: -->
<link
  rel="preload"
  as="image"
  href="{{ section.settings.hero_image | image_url: width: 1200 }}"
  fetchpriority="high"
>

Convertir imágenes a formato WebP
WebP es hasta 80% más liviano que JPG/PNG sin pérdida visual perceptible

Node.js (Sharp)Copiar
// npm install sharp
const sharp = require('sharp');
const path = require('path');

async function convertToWebP(inputPath) {
  const outputPath = inputPath.replace(/\.(jpg|jpeg|png)$/i, '.webp');
  await sharp(inputPath)
    .webp({ quality: 82, effort: 6 })
    .toFile(outputPath);
  console.log(`✓ Converted: ${path.basename(outputPath)}`);
}

// Conversión masiva de una carpeta
const glob = require('glob');
glob('./public/images/**/*.{jpg,jpeg,png}', (err, files) => {
  files.forEach(convertToWebP);
});

🧹 CSS & JavaScript

Scripts y estilos mal cargados bloquean el render de la página. Estas técnicas eliminan el render-blocking y reducen el trabajo del navegador.

52
score actual
MEJORAR
⏳
Defer y Async en scripts no críticos
Evita que JS bloquee la renderización del contenido visible
Alto impacto

Sin defer o async, cada script detiene completamente el render de la página mientras se descarga y ejecuta. Analytics, chat widgets, pixels de Meta y Google son los principales culpables. Diferirlos puede reducir el LCP hasta 1.5 segundos.

HTML — Script loading strategyCopiar
<!-- ❌ ANTES: bloquea el render -->
<script src="/js/analytics.js"></script>

<!-- ✅ DEFER: descarga en paralelo, ejecuta después del HTML -->
<script src="/js/analytics.js" defer></script>

<!-- ✅ ASYNC: descarga en paralelo, ejecuta apenas esté listo -->
<script src="/js/chat-widget.js" async></script>

<!-- Pixel de Meta: retraso hasta interacción del usuario -->
<script>
  function loadMetaPixel() {
    if (window._fbLoaded) return;
    window._fbLoaded = true;
    // ... código del pixel aquí
  }
  ['scroll', 'click', 'touchstart'].forEach(e =>
    window.addEventListener(e, loadMetaPixel, { once: true })
  );
</script>
✂️
Critical CSS inline (above the fold)
Renderiza el contenido visible sin esperar el CSS completo
Impacto medio

El Critical CSS extrae solo los estilos necesarios para renderizar lo visible al cargar, los incluye inline en el <head>, y carga el CSS completo de forma diferida. El resultado: la página parece cargarse instantáneamente.

HTML — Patrón Critical CSSCopiar
<head>
  <!-- 1. Critical CSS inline (extraído con la herramienta de abajo) -->
  <style>
    /* Solo estilos visibles en el primer viewport */
    body{margin:0;font-family:sans-serif}
    .header{background:#000;color:#fff;padding:16px}
    .hero{min-height:60vh;display:flex;align-items:center}
    /* ... (extraer con: npx critical URL --inline) */
  </style>

  <!-- 2. CSS completo cargado de forma no bloqueante -->
  <link
    rel="preload"
    href="/css/main.css"
    as="style"
    onload="this.onload=null;this.rel='stylesheet'"
  >
  <noscript>
    <link rel="stylesheet" href="/css/main.css">
  </noscript>
</head>

🔤 Optimización de tipografías

Las fuentes de Google Fonts, Typekit o similares agregan requests externos que aumentan el TTFB. Self-hosting y preload eliminan esta latencia completamente.

61
score actual
MEJORAR
📥
Self-hosting de fuentes + font-display: swap
Elimina la dependencia de Google Fonts y evita el FOUT (Flash of Unstyled Text)
Impacto medio

Cada llamada a Google Fonts requiere una conexión DNS + TCP adicional (~200ms de latencia). Hospedar las fuentes en tu propio servidor y usar font-display: swap muestra texto inmediatamente con la fuente del sistema mientras carga la tuya — eliminando el bloqueo visual.

CSS — @font-face optimizadoCopiar
/* 1. Descargar fuentes desde: https://gwfh.mranftl.com/fonts */
/* 2. Subir archivos .woff2 a tu servidor */
/* 3. Reemplazar @import de Google Fonts con: */

@font-face {
  font-family:   'TuFuente';
  src:           url('/fonts/tu-fuente-regular.woff2') format('woff2');
  font-weight:   400;
  font-style:    normal;
  font-display:  swap; /* Muestra texto inmediatamente */
}

@font-face {
  font-family:   'TuFuente';
  src:           url('/fonts/tu-fuente-bold.woff2') format('woff2');
  font-weight:   700;
  font-style:    normal;
  font-display:  swap;
}

📱 Optimización móvil

El 73% de compras en ecommerce latinoamericano se inician desde móvil. Una experiencia móvil deficiente es ventas directamente perdidas.

👆
Touch targets & botones accesibles
Botones CTA con área táctil mínima de 44×44px (estándar WCAG 2.5.5)
Alto impacto

Botones pequeños en móvil generan toques erróneos y frustración. El estándar WCAG requiere 44×44px mínimo de área táctil. En Shopify y ecommerce, el botón "Agregar al carrito" es el elemento más crítico — si es difícil de tocar, pierdes la conversión.

CSS — Botones táctiles optimizadosCopiar
/* Estándar mínimo para todos los elementos clicables en móvil */
.btn, button, a, [role="button"] {
  min-height:      44px;
  min-width:       44px;
  padding:         12px 20px;
  touch-action:   manipulation; /* Elimina delay de 300ms en iOS */
  -webkit-tap-highlight-color: transparent;
}

/* Botón CTA principal: grande y visible */
.btn-add-to-cart {
  width:           100%;
  min-height:      52px;
  font-size:       16px; /* ≥16px evita zoom automático en iOS */
  font-weight:     600;
  border-radius:   10px;
}

/* Tipografía mínima: ≥14px en móvil */
@media (max-width: 768px) {
  body { font-size: 15px; }
  p, li, td { font-size: 14px; line-height: 1.65; }
